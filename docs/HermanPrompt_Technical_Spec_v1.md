# HermanPrompt Technical Specification v1

## 1. Purpose

HermanPrompt is a ChatGPT-like chat shell that routes user instructions through Prompt Transformer before those instructions are executed by the configured LLM.

The product should feel like a familiar modern chat app first, while making Prompt Transformer visible only when the user explicitly chooses to inspect it.

## 2. Product Definition

HermanPrompt is not just a prompt testing screen. It is a chat application shell intended to simulate the feel of a mainstream LLM chat experience while routing user instructions through Prompt Transformer.

### Product statement

HermanPrompt is a ChatGPT-like chat application that sends user instructions through Prompt Transformer before execution by the LLM.

### Product principle

HermanPrompt should behave like a normal premium chat application first.

Prompt Transformer is a middleware layer in the request path, not the primary surface of the product.

## 3. Goals

### In scope for v1

- ChatGPT-like single-thread chat shell
- Text-only messaging for the initial MVP
- `user_id_hash` query parameter initialization
- theme initialization by query parameter
- backend orchestration from UI -> HermanPrompt backend -> Prompt Transformer -> configured LLM
- inline "Show Details" behavior for transformed prompts
- response-level feedback collection
- responsive layout suitable for iframe embedding
- deployment on Railway from GitHub

### Out of scope for v1

- persistent conversation history
- loading previous conversations
- user authentication
- profile editing
- file upload in v1
- voice or multimodal input
- agent workflows

## 4. Product Behavior

### Core flow

1. The app loads inside a browser or iframe.
2. The frontend reads `user_id_hash` from the URL query string.
3. The user enters a message.
4. The frontend sends the message and conversation context to the HermanPrompt backend.
5. The backend calls the Prompt Transformer service to rewrite the user instruction layer.
6. The Prompt Transformer returns a deterministic corrected prompt.
7. The HermanPrompt backend sends the transformed instruction plus the active chat context to the configured LLM.
8. The backend returns the assistant response and metadata to the frontend.
9. The frontend renders the exchange inline in the transcript.
10. The user can optionally inspect details or submit feedback on the response.

### Chat shell behavior

The MVP should feel like a standard chat product:

- messages appear as transcript turns
- a compact in-progress indicator appears during response generation
- the transcript scrolls as new content is added
- the header remains fixed while the chat area scrolls
- advanced details remain optional

### Show Details behavior

The transcript is rendered as a sequence of chat items in this order:

1. Original user prompt
2. Transformed prompt, only when "Show Details" is enabled
3. Assistant response

The transformed prompt is displayed inline in the same transcript flow, not in a side panel, modal, or debug drawer.

When "Show Details" is disabled, transformed prompt rows are hidden from the transcript, but the backend still performs prompt transformation.

The toggle is global for the current browser session and affects the visibility of transformed prompt rows for all turns currently loaded in memory.

### History behavior

Persistent history storage is a future enhancement.

For v1:

- the current conversation exists only in frontend memory
- refreshing the page clears the visible transcript unless the browser preserves state
- the backend does not provide history retrieval endpoints
- `conversation_id` is still generated by the frontend and sent with each request so turns can be grouped for analytics and future compatibility

### Future chat-shell behaviors

These are explicitly planned after the MVP foundation is stable:

- streaming assistant responses
- retry or regenerate
- copy response
- persistent conversation history
- attachment support

### Guide Me coaching flow

Guide Me is a guided prompt-construction and repair flow, not a freeform coaching chat.

Its purpose is to help the user assemble a prompt that passes Prompt Transformer requirements and is strong enough to target a `100/100` validated result.

Architectural boundary:

- Prompt Transformer evaluates and scores
- Herman Prompt displays and guides

Herman Prompt must not create a separate scoring model for Guide Me.

Guide Me should:

1. ingest the user's original prompt when available
2. parse existing labeled fields already present in that prompt
3. validate and score the original prompt against Prompt Transformer
4. identify only the elements that do not receive maximum points
5. ask targeted questions only for those non-max elements
6. merge each user answer across all relevant prompt sections
7. re-score after each change and continue only on the remaining weak elements
8. compile a final labeled prompt
9. switch into specificity mode once structure is complete
10. keep refining one remaining issue at a time
11. complete only when the prompt reaches the practical stop threshold

#### Guide Me step model

- `Prompt Intake`
  Purpose: ingest and score the user's original prompt before asking anything else
- `Intro`
  Purpose: confirm whether the user's usual AI use matches today's need
- `Describe Need`
  Purpose: capture today's task when the intro answer is `No` or when the original prompt does not provide enough task information
- `Who`
  Purpose: improve the role or persona only if `Who` is below the maximum score
- `Why`
  Purpose: improve instructions, guardrails, and preferences only if more direction is still needed
- `How`
  Purpose: improve context and background only if `Context` is below the maximum score
- `What`
  Purpose: improve delivery format and structure only if `Output` is below the maximum score
- `Refine`
  Purpose: repair a specific below-max section or score gap using prompt-ready examples only
- `Complete`
  Purpose: present a validated compiled prompt for insertion into the composer

Intro routing rule:

- if the user answers `No`, the wizard must go to `Describe Need`
- if the user answers `Yes`, the wizard must advance to the next actually needed collection step
- the intro response must not trigger prompt validation or completion

#### Final prompt format

In full enforcement mode, Guide Me must output explicitly labeled sections:

- `Who:`
- `Task:`
- `Context:`
- `Output:`
- `Additional Information:`

#### Refinement rules

- Prompt Transformer must be the source of truth for section scoring and requirement status.
- Herman Prompt may route the wizard, but it must do so from transformer-provided field results.
- The original prompt must be the starting point for the entire repair flow.
- The current conversation context should also be used when interpreting the original prompt.
- Sections that already have maximum points must not be asked again.
- Normal collection steps should not duplicate helper text in a second coaching panel.
- Any example shown in the wizard should be professional and tailored to the user's prompt context.
- If validation still fails, the flow should return to the exact weak section instead of looping through a generic refine screen.
- Refinement must be tied to the currently weak element or explicit scoring gap.
- Refinement options must be phrased as prompt-ready text that can be inserted directly into the final prompt.
- Contextual refinement options should be AI-generated from the current compiled prompt, transformer `reason`, transformer `improvement_hint`, and current score state.
- Hard-coded refinement options should be retained only as a fallback if the AI-generated options fail validation or cannot be produced.
- Refinement must not repeat as a generic polish loop once the actual weak area has been addressed.
- Once structure is complete, the wizard should stay in `Refine` and should not route back into section prompts.
- If the prompt is structurally complete but still below threshold, refinement should explain the remaining specificity gap in plain language instead of sending the user back through collection prompts.
- Freeform refinements typed by the user must be treated as prompt updates, not only as numbered option selections.
- Numbered option selection should only occur when the user's answer is actually a list of option numbers.
- Task-aware examples and fallback suggestions must reflect the user's domain and task instead of generic "subject-matter expert" copy.
- If a refinement produces no score delta, the next refinement pass must change strategy instead of reissuing the same idea.
- The practical stop threshold is `95/100` overall and, when available, `95/100` for LLM score.

#### Specificity mode

Specificity mode begins after transformer requirements indicate that `Who`, `Task`, `Context`, and `Output` are all present.

Specificity mode should:

- focus on one remaining issue only
- prefer a real weak field when transformer scoring identifies one
- otherwise infer one whole-prompt specificity gap
- keep the user in `Refine` until the prompt reaches the stop threshold
- avoid re-asking collection questions for already-complete sections
- keep the guidance and the suggested rewrites aligned to that same issue

#### Debugging support

Guide Me debugging support should include:

- requirement debug payload
- decision trace payload
- unit tests for decision logic
- smoke tests for service-level flow
- local scenario runner for repeatable end-to-end prompt cases

The scenario runner should capture:

- baseline score before Guide Me
- focus field per step
- suggestions shown
- applied refinement
- score delta after each step
- repeated-issue detection

#### Prompt Transformer API Requirements

For Guide Me to stay aligned with the main scoring model, Prompt Transformer should return field-level scoring details for each prompt element.

Required per-field response shape:

```json
{
  "conversation": {
    "requirements": {
      "who": {
        "status": "present",
        "heuristic_score": 25,
        "llm_score": 22,
        "max_score": 25,
        "reason": "Role is specific.",
        "improvement_hint": null
      },
      "task": {
        "status": "derived",
        "heuristic_score": 14,
        "llm_score": 12,
        "max_score": 25,
        "reason": "Task is too broad.",
        "improvement_hint": "State the exact outcome."
      },
      "context": {
        "status": "present",
        "heuristic_score": 25,
        "llm_score": 24,
        "max_score": 25,
        "reason": "Context is clear.",
        "improvement_hint": null
      },
      "output": {
        "status": "present",
        "heuristic_score": 20,
        "llm_score": 18,
        "max_score": 25,
        "reason": "Output is defined but not precise enough.",
        "improvement_hint": "Add exact structure and length."
      }
    }
  }
}
```

Herman Prompt should consume these fields directly and stop displaying local heuristic substitutes once Prompt Transformer supports them.

## 7. Runtime Architecture

### High-level components

- frontend web client
- HermanPrompt backend API
- Prompt Transformer service
- LLM provider API
- HermanPrompt database for feedback storage

### Architectural definition

HermanPrompt is a chat shell with Prompt Transformer in the middle.

That means:

- the frontend behaves like a regular chat client
- the backend owns orchestration, context assembly, and provider execution
- Prompt Transformer rewrites the user instruction layer
- the LLM still produces the final assistant response

### Request path

1. Frontend calls HermanPrompt backend `POST /api/chat/send`.
2. HermanPrompt backend validates the request.
3. HermanPrompt backend constructs the current request context.
4. HermanPrompt backend calls Prompt Transformer `POST /api/transform_prompt` for the current user instruction.
5. Prompt Transformer returns `transformed_prompt` and metadata.
6. HermanPrompt backend calls the configured LLM with the transformed instruction and current conversation context.
7. HermanPrompt backend returns transcript payload data to the frontend.
8. Frontend renders the original prompt, optional transformed prompt, and assistant response.

### Future-ready architectural direction

The backend contract should evolve toward a chat-oriented model rather than a single-string prompt model.

Future request shape should support:

- conversation messages
- optional attachments
- transformed instruction layer
- assistant output
- optional debug metadata

The current MVP may still submit a single `message_text`, but the architecture should be treated as chat-native, not prompt-form-native.

## 8. Frontend Specification

### Experience goal

The frontend should feel close to a mainstream chat application, not like an admin console or developer harness.

### Query parameters

Required:

- `user_id_hash`

Optional:

- `show_details=true|false`
- `theme=dark|light`

Frontend behavior:

- if `user_id_hash` is missing, render a blocking error state and disable chat submission
- `show_details` initializes the toggle state when provided
- if `show_details` is absent, default to `false`
- `theme` initializes the visual theme when provided
- if `theme` is absent, default to `light`
- values outside `dark|light` fall back to `light`

### Main screens and states

The app is a single-screen chat interface with these states:

- empty state
- active conversation
- request in progress
- submission error
- feedback modal open

### UI components

#### Top bar

- HermanPrompt branding
- Show Details toggle
- theme-aware styling that preserves contrast in dark and light modes

#### Transcript

Each user turn may render:

- `user` message row
- `transformed_prompt` message row, conditionally visible
- `assistant` message row

The transformed prompt row must be visually distinct from both user and assistant messages and labeled clearly, for example: `Transformed Prompt`.

#### Composer

- multiline text input
- send button
- Enter submits
- Shift+Enter inserts newline

#### Feedback controls

Each assistant message includes:

- thumbs up icon button
- thumbs down icon button

Selecting either opens a modal.

### Visual direction

The visual baseline should follow the provided mockup:

- a clean chat window layout with a slim top bar
- brand logo left-aligned in the header
- primary controls right-aligned in the header
- large open transcript area with minimal chrome
- a rounded bottom-docked composer with subtle elevation

The design should feel intentional and polished, closer to a modern chat product than an internal admin tool.

### Theme requirements

The app must support both dark and light themes.

Dark theme should reflect the mockup as the default visual direction:

- charcoal or graphite background
- slightly elevated surfaces for header and composer
- muted borders
- bright, readable text

Light theme should preserve the same layout and hierarchy while shifting to:

- soft light neutral background
- white or near-white elevated surfaces
- subtle gray borders
- dark readable text

The implementation should use CSS custom properties or an equivalent token system so both themes share the same component structure and spacing.

### Message presentation

The transcript should visually separate three message types:

- user messages
- transformed prompt messages
- assistant messages

Recommended behavior:

- user messages align to the right or use a visually distinct surface
- assistant messages align to the left or use a separate neutral surface
- transformed prompt messages appear between them with a labeled system-style treatment

The transformed prompt row should be clearly identifiable as an intermediate system step, not as a user or assistant message.

### Header behavior

The header must include:

- HermanPrompt branding on the left
- `Show Details` toggle on the right

Optional for v1:

- a theme indicator or lightweight label

Theme selection itself is controlled by the URL query parameter in v1, not by an in-app toggle.

### Composer behavior

The composer should match the mockup direction:

- anchored near the bottom of the viewport
- rounded rectangular container
- multiline text area
- primary send affordance on the right

Icons other than send are not required for v1 and should not imply unsupported features such as voice, file upload, or web search unless those features are implemented.

### Chat fidelity requirements

The frontend should preserve familiar chat-app interaction patterns:

- the header stays pinned
- the transcript owns scrolling
- new content scrolls into view
- response generation shows lightweight activity without interrupting the layout
- transformed prompt visibility is optional and off by default

### Accessibility requirements

- keyboard accessible composer, toggle, and feedback controls
- visible focus states
- sufficient color contrast
- button labels and modal inputs must be screen-reader accessible

### Responsive requirements

- layout must work at desktop width and mobile iframe widths
- no horizontal overflow in transcript or composer
- composer remains usable on narrow screens
- theme styling must remain legible and visually coherent at all supported widths

## 9. Backend Specification

### Responsibilities

The HermanPrompt backend is the orchestration layer. It is responsible for:

- validating frontend requests
- calling the Prompt Transformer
- assembling the current request context
- calling the configured LLM
- formatting the response payload for the frontend
- storing feedback submissions
- hiding all provider API keys and service secrets from the browser

### Non-responsibilities

The HermanPrompt backend is not responsible for:

- profile generation
- prompt transformation logic
- persistent conversation history in v1

### Architectural role

The backend should be implemented as a chat orchestration service, not as a narrow prompt relay.

Even if the MVP request body is small, the backend should be built so it can later support:

- chat history
- attachments
- streaming
- richer provider tool usage

## 10. API Specification

### 10.1 Health endpoint

`GET /api/health`

Response:

```json
{
  "status": "ok"
}
```

### 10.2 Send chat turn

`POST /api/chat/send`

Request body:

```json
{
  "user_id_hash": "user_1",
  "conversation_id": "conv_2d6b7c57",
  "message_text": "Explain this concept simply"
}
```

Request rules:

- `user_id_hash` is required
- `conversation_id` is required
- `message_text` is required and non-empty after trimming

MVP note:

The current request shape is intentionally narrow for the initial build, but it should be treated as the first version of a broader chat contract.

Backend orchestration:

1. Call Prompt Transformer with:

```json
{
  "session_id": "conv_2d6b7c57",
  "user_id": "user_1",
  "raw_prompt": "Explain this concept simply",
  "target_llm": {
    "provider": "openai",
    "model": "gpt-4.1"
  }
}
```

2. Use `transformed_prompt` as the transformed instruction layer for the LLM request.
3. Include current in-memory conversation context when supported by the backend implementation.
4. Return frontend-oriented response payload.

Successful response:

```json
{
  "conversation_id": "conv_2d6b7c57",
  "turn_id": "turn_8f2dd2b4",
  "user_message": {
    "role": "user",
    "text": "Explain this concept simply"
  },
  "transformed_message": {
    "role": "transformed_prompt",
    "text": "Explain the topic according to the guidance below.\n..."
  },
  "assistant_message": {
    "role": "assistant",
    "text": "Here is a simple explanation..."
  },
  "metadata": {
    "transformer": {
      "task_type": "explanation",
      "persona_source": "db_profile",
      "profile_version": "v1",
      "requested_model": "gpt-4.1",
      "resolved_model": "gpt-4.1",
      "used_fallback_model": false,
      "rules_applied": [
        "task:explanation:keyword",
        "persona:answer_first:enabled"
      ]
    },
    "llm": {
      "provider": "openai",
      "model": "gpt-4.1"
    }
  }
}
```

Notes:

- `transformed_message` is always returned by the backend even if the frontend toggle is off
- the frontend decides whether to render `transformed_message`
- `turn_id` is generated by the backend and is used for feedback submission

### 10.3 Submit feedback

`POST /api/feedback`

Request body:

```json
{
  "turn_id": "turn_8f2dd2b4",
  "conversation_id": "conv_2d6b7c57",
  "user_id_hash": "user_1",
  "feedback_type": "down",
  "selected_dimensions": [
    "directness",
    "ambiguity_reduction"
  ],
  "comments": "Still too vague."
}
```

Rules:

- `feedback_type` must be `up` or `down`
- `selected_dimensions` may contain one or more values from the approved list
- `comments` is optional

Response:

```json
{
  "status": "saved"
}
```

### 10.4 Approved feedback dimensions

- `structure`
- `answer_position`
- `directness`
- `level_of_detail`
- `ambiguity_reduction`
- `exploration_level`
- `supporting_context`

### 10.5 Future contract direction

The next contract revision should move toward a chat-native request model, for example:

```json
{
  "user_id_hash": "user_1",
  "conversation_id": "conv_2d6b7c57",
  "messages": [
    {
      "role": "user",
      "text": "Explain this concept simply"
    }
  ],
  "attachments": [],
  "debug": {
    "show_details": false
  }
}
```

The backend may still transform only the most recent user instruction, but the overall contract should evolve around messages and chat state rather than a single raw prompt string.

## 11. Prompt Transformer Integration

HermanPrompt must integrate with the existing Prompt Transformer contract in `/Users/michaelanderson/projects/prompt_transformer/docs/api_contract.md`.

### Transformer endpoint

`POST {PROMPT_TRANSFORMER_URL}/api/transform_prompt`

### HermanPrompt mapping to transformer contract

- HermanPrompt `conversation_id` -> transformer `session_id`
- HermanPrompt `user_id_hash` -> transformer `user_id`
- HermanPrompt `message_text` -> transformer `raw_prompt`
- HermanPrompt configured LLM provider/model -> transformer `target_llm`

### Transformer errors

- `400` from transformer should be surfaced as a client-safe HermanPrompt error
- `503` from transformer should become a temporary backend error message
- timeouts should be treated as upstream service failure

## 12. LLM Provider Integration

### v1 behavior

- one LLM provider is configured by environment variables
- one model is configured by environment variables
- the frontend cannot choose provider or model
- the transformed instruction layer is used in the LLM request path

### Initial target

OpenAI-compatible Responses API integration is the recommended initial path.

### Minimum backend abstraction

Create a provider service interface so the backend can support future providers without changing route logic.

Suggested module:

- `backend/app/services/llm_client.py`

### Future provider capability roadmap

The architecture should leave room for later support of:

- attachments
- tool-enabled responses
- streaming output

Provider support may be intentionally limited to models and APIs compatible with the OpenAI Responses-style contract.

## 13. Data Model

### v1 persistence scope

Persist only feedback records in HermanPrompt v1.

Do not persist:

- full conversation transcripts
- user message history
- transformed prompts for replay
- assistant responses for history retrieval

### Feedback table

`feedback`

Fields:

- `id` UUID primary key
- `turn_id` string, indexed
- `conversation_id` string, indexed
- `user_id_hash` string, indexed
- `feedback_type` string
- `selected_dimensions` JSON or Postgres array
- `comments` text nullable
- `created_at` timestamp

## 14. Error Handling

### Frontend-visible errors

- missing `user_id_hash`: block chat and show configuration error
- empty message: inline validation error
- Prompt Transformer unavailable: show retryable error
- LLM provider failure: show retryable error
- feedback save failure: show non-blocking toast or inline message

### Backend HTTP behavior

- invalid request payload: `400`
- upstream transformer validation error: `400`
- upstream transformer unavailable: `502` or `503`
- LLM provider failure: `502`
- unexpected server error: `500`

The backend should not leak raw provider stack traces or secrets in error responses.

## 16. Security and Embedding

### Security requirements

- no PII is stored by HermanPrompt
- `user_id_hash` is treated as an opaque identifier
- all secrets remain server-side
- the frontend must never call the LLM provider directly

### Embedding requirements

- the frontend must render correctly inside an iframe
- the backend must allow the deployed frontend origin via CORS
- if the parent site origin is known, configure iframe-related headers accordingly

### v1 assumption

The app is an internal demo/test tool, so authentication is omitted in v1. If it becomes externally reachable, auth and rate limiting must be added before broader release.
